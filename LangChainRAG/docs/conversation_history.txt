Conversation History
=====================

Conversation history refers to the record of all messages exchanged between a user and an AI model (or agent) during a conversation. It is the primary mechanism for maintaining context within a session and is fundamental to how modern LLMs operate in chat mode.

How Conversation History Works with LLMs
-----------------------------------------

LLMs are stateless — they do not inherently remember previous interactions. Every API call is independent. To create the illusion of a continuous conversation, the entire conversation history is sent with each request.

A typical conversation history looks like:
[
  {"role": "system", "content": "You are a helpful assistant."},
  {"role": "user", "content": "What is RAG?"},
  {"role": "assistant", "content": "RAG stands for Retrieval-Augmented Generation..."},
  {"role": "user", "content": "How does it reduce hallucination?"},
  {"role": "assistant", "content": "By grounding responses in retrieved facts..."}
]

Message Roles:
- system: Sets the behavior, personality, and constraints of the assistant. Sent once at the beginning.
- user: Messages from the human user.
- assistant: Responses generated by the AI model.
- tool: Results from tool calls, providing external data to the model.

Why Conversation History Matters:
1. Context Continuity: The model understands follow-up questions ("How does it reduce hallucination?" makes sense only with the prior RAG discussion).
2. Coreference Resolution: The model can resolve pronouns and references ("it" refers to "RAG" from the earlier message).
3. Task Memory: The model remembers what has already been done in a multi-step task.
4. Personality Consistency: The system message is reinforced throughout the conversation.

Conversation History Management Strategies
--------------------------------------------

Full History:
- Send the entire conversation every time.
- Simple but becomes expensive and eventually exceeds the context window.
- Works well for short conversations (under ~20 turns).

Truncation:
- Remove the oldest messages when approaching the context limit.
- Always keep the system message.
- Simple but loses important early context.

Sliding Window:
- Keep the most recent N messages.
- Optionally include a summary of dropped messages.
- Balances recency with context preservation.

Summarization:
- Periodically summarize the conversation history and replace older messages with the summary.
- Preserves the essence of past exchanges while saving tokens.
- Example: After every 10 messages, generate a summary and replace messages 1-10 with the summary.

Selective Memory:
- Identify and retain only the most important messages (key decisions, user preferences, critical context).
- Use heuristics or an LLM to determine message importance.

Token Counting:
- Track token usage precisely using tokenizers (e.g., tiktoken for OpenAI models).
- Trim messages from the oldest when the total exceeds a threshold (e.g., 80% of the context window).

Multi-Turn Conversation Challenges:
- Token Cost: Each message is re-sent with every API call, so long conversations become expensive.
- Context Window Limits: Even large context windows (128K+ tokens) can be exceeded in extended sessions.
- Lost Context: Important early context may be truncated in long conversations.
- Latency: Larger prompts take longer to process.
- Consistency: The model may contradict earlier statements if key context was dropped.

Conversation History in RAG Systems:
In RAG applications, conversation history plays a dual role:
1. It provides conversational context for the LLM's response generation.
2. The current question (informed by history) is used to generate a search query for retrieval.

A common pattern is query reformulation: before searching the knowledge base, rewrite the user's latest message as a standalone question that incorporates context from the conversation history.

Example:
- History: "Tell me about ChromaDB" → "What distance metrics does it support?"
- Reformulated query: "What distance metrics does ChromaDB support?"
This ensures the retrieval system gets a complete, unambiguous query.
